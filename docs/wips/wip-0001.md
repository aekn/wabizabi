## Abstract

This document proposes Wabizabi, a runtime for orchestrating AI systems in Python.

The project is intentionally placed below the abstractions that many current frameworks expose by default. The goal is not to define what an agent is, what modalities are in play, or how people should structure an application. The goal is to define a runtime where computation, state, capabilities, and recorded execution are explicit enough that durability, replay, observability, and operator control can come from the runtime itself rather than being added as separate systems later.

This WIP is the architectural overview. It explains the design pressures that led to the project, states the main commitments I am making, summarizes the primitive concepts the runtime is built around, defines the intended boundary between core, standard library, and extensions, and records the non-goals that shape the v1 intention. Later WIPs specify the details, but this document is meant to stand on its own as the architectural overview.

## Motivation

The main question behind Wabizabi is simple. What should the core runtime of an AI system actually be built around?

A lot of current tools start from a user-facing abstraction such as an agent, a graph, a flow, or a program module. That is often useful. It gives people something concrete to author right away. I am trying a different starting point. I want to build the runtime around the mechanics that seem to show up no matter what the application pattern is: computation, state, external, capabilities, and recorded execution.

I got to that position by running into the same three problems repeatedly.

The first problem is that description and execution are often too close together. In many systems, the thing you author is also the thing you call to run. That is convenient, but it mixes two different ideas. One is a description of what should happen. The other is the runtime handle that actually makes it happen. A description should be easy to inspect, serialize, diff, replay, or send somewhere else. Execution is where effects happen. It depends on bindings, state, retries, timeouts, concurrency, durability, and the rest of the runtime environment. When those two ideas are too tightly fused, the system can still support replay, inspection, or remote execution, but those features tend to feel like features attached to the runtime object rather than consequences of a clean description layer.

The second problem is that durable execution often feels like a separate path. In some systems, persistence and recovery are introduced through a checkpointer, a deployment surface, or a more operational execution mode. That is not inherently wrong. In many systems it is the practical choice. The problem is that it becomes easy for the normal execution path and the durable execution path to drift in behavior or in how people think about the system. Durability starts to feel like an add-on instead of part of the runtime contract.

The third problem is that model calls, tool dispatch, database access, and other external interactions are often written directly inside orindary functions or methods. That is flexible, and I do not think flexibility is the problem. The issue is that once the runtime boundary is implicit, it becomes harder to apply retry, caching, fallback, idempotency, auditing, and substitution in one clear place. Those policies still have to exist, but they tend to be repeated locally or wrapped in ad hoc helpers.

Wabizabi is an attempt to respond to those three problems directly. It keeps description and execution separate. It treats durable execution as a scheduler guarantee rather than a different authoring style. It makes external capabilities explicit at the runtime boundary so middleware has one place to operate.

I do not think these observations prove that current frameworks are wrong, and I do not think they prove that Wabizabi will necessarily age better. What they do is motivate the architecture I want to try: one where computation, state, capabilities, and recorded execution are first-class primitives, and where agents, pipelines, and other higher-level patterns are built from those primitives rather than embedded into the core.

## Design Goals

This project is trying to satisfy a small set of goals at the same time.

The first goal is to keep the runtime model smaller than the set of application patterns people may eventually build on top of it. I do not want core to start with "agent", "graph", or "flow" as its main concept and then work backward from there.

The second goal is to make the boundary between logic and external capabilities explicit. A model call, a tool registry, or a database should all be things the runtime can see and mediate.

The third goal is to make durable execution a scheduler guarantee, not a different programming model. The same authored computation should run under a low-latency scheduler, a durable scheduler, or a replay scheduler without the user having to rewrite it for each mode.

The fourth goal is to keep authoring close to ordinary Python. I do not want a workflow DSL. I want users to write `if`, `for`, `await`, `parallel(...)`, and small helper abstractions in Python.

The fifth goal is to keep the line between core, standard library, and extensions strict enough that change mostly lands outside the primitive layer. I do not think core will never need revision. I do think it should change much less often than providers, protocol adapters, content formats, and convenience APIs do.

## Core Commitments

Three commitments shape the rest of the design.

### Durable execution is based on recorded execution history

For executions that require recovery, audit, and replay, Wabizabi treats the recorded execution history as the source of truth. The practical meaning of that is straightforward. If an execution needs strong recovery and replay guarantees, it should run on `DurableScheduler`. The scheduler records execution history to a `Journal`, and replay and recovery work from that recorded history. To clarify, this does not mean that every trace from every scheduler mode is equally complete.

### Capabilities are explicit runtime objects

A model provider, tool registry, database, or other external dependency should not be hidden from the runtime. In Wabizabi, these are explicit capabilities that an Op declares in its signature and the scheduler resolves from the current scope.

This matters because it gives the runtime a place to apply retry, caching, rate limiting, fallback, idempotency, auditing, and substituion. It also means testing and production configuraiton use the same mechanism. Replacing a real model provider with a mock should not need a separate testing system.

### Higher-level abstractions should share the same runtime

I do not want one runtime for agents, another for workflows, and another for everything else.

If the standard library ships an `Agent`, that should still turn into the same runtime concepts used everywhere else. An agent call should still be an Op invocation. Tool dispatch should still go through the same capability boundary. State should still live in Cells. Durability should still come from the scheduler and journal rather than from a separate agent-specific system.

This matters because convenience APIs can easily become the real runtime in practice. I want the standard library to be useful, but I do not want it to replace the primitive model underneath.

## The Five Primitives

Wabizabi is built around five primitives.

### `Op`

An `Op` is a unit of computation as inert, content-addressed data.

An Op is not a running Python frame. It is a record that describes a function body, its schema, its declared capabilities, and its identity. Calling an Op constructs an `OpInvocation`. Execution happens when a scheduler evaluates that invocation.

This distinction between description and execution is one of the main design choices in Wabizabi.

### `Cell`

A `Cell` is named, versioned state.

A Cell is not just a mutable value with logging attached. Every write creates a new immutable version. The current value is the latest verison. Older versions remain in history unless they are explicitly garbage-collected.

This is the intended state model because it allows for replaying, time-travel inspection, branching, and durability.

### `Stream`

A `Stream` is a bounded channel for values that are consumed as they arrive.

Cells model named, versioned state. Streams model ordered values in transit. Token streams, progress updates, audio frames, event feeds, and similar cases belong here.

### `Effect`

An `Effect` is a named capability at the `Op` boundary.

Effects specify what an `Op` requires from outside itself. This includes models, tools, browsers, stores, databases, and user-defined capabilities.

The important point is that effects are visible to the runtime, which can mediate them and enforce policy around them.

### `Scheduler`

A `Scheduler` is the execution kernel.

The `Scheduler` evaluates `Op`s, resolves `Effect`s, manages `Cell`s and `Stream`s, emits the execution record, and mediates pause, resume, checkpointing, and replay.

The same authored computation should run on all `Scheduler` variants. What changes is the guarantee, not the authoring model.

## Architectural Shape

A few consequences follow from those primitives.

### Description and execution are separate

An `Op` describes a computation. An `Op` invocation describes an intended call. The scheduler executes it.

That split is what makes serialization, remote dispatch, replay, structural diffing, and stepwise continuation part of the architecture instead of things added after the fact.

### State is versioned, not ambient

A `Cell` is runtime state with identity and history. This means replay and durability can talk about the same object model that the application talks about.

### Capabilities are resolved through scope

An `Op` does not directly construct the model, tool registry, or database it uses. It declares the capability. The scheduler resolves the current binding from the current scope. Child scopes can override bindings for a subtree of execution.

This is how model swaps, tenant-specific bindings, budgeted overrides, and test substitutions all become the same operation instead of separate systems.

### Recovery and replay are scheduler concerns

`Op`s should not need a "durable version" and a "non-durable version." The scheduler mode decides what guarantees exist around the same authored computation.

### Middleware belongs at the capability boundary

Retry, caching, fallback, idempotancy, rate limiting, cost tracking, and auditing should be written once at the capability boundary rather than repeated inside every `Op` body.

## Scheduler Modes

Wabizabi has three reference scheduler modes.

### `HotScheduler`

`HotScheduler` is the low-latency mode. It is aimed at interactive use and best-effort tracing. It may export trace data and support inspection, but under load it may drop events or otherwise provide weaker guarantees than durable mode.

### `DurableScheduler`

`DurableScheduler` is the mode that carries the full durability contract. It persists the execution record to a journal and uses that record as the basis for recovery, replay, and operator control across process death.

### `ReplayScheduler`

`ReplayScheduler` is the mode that treats a past durable execution record as a first-class artifact that can be re-executed for audit, debugging, regression checking, and time-travel inspection.

## Layering and Boundaries

Wabizabi is organized into three layers.

### Core (`wabizabi.core`)

Core contains the primitives and their contracts:

- `Op`
- `Cell`
- `Stream`
- `Effect`
- `Scheduler`
- the trace event schema
- journal and exporter protocols
- a small set of runtime-aware concurrency helpers

Core does not define agents, prompt builders, tool surfaces, evaluation policy, or provider-specific behavior.

### Standard library (`wabizabi.<domain>`)

Standard library contains reference implementations and common patterns built from core. This includes public effect protocols, provider handlers, stores, journals, exporters, middleware, tools and tool registries, memory helpers, prompt construction, testing helpers, codec adapters, an evaluation surface, and a standard library `Agent`.

The point of the standard library is not to expand the core. It is to ship patterns that many users will otherwise rebuild for themselves, each in slightly different ways.

### Extensions (`wabizabi.ext.<vendor>`)

Extensions live under `wabizabi.ext.*` and are for protocols Wabizabi does not own. MCP, FastAPI, Slack, GitHub, Playwright, and similar systems belong here.

The division of responsibility is as follows. Core defines mechanisms and contracts. Standard library provides common policies built on those contracts. Extensions adapt external protocols to those contracts. As a consequence, most ecosystem change should land in standard library and extensions rather tahn in core.

## Prior Art

The most direct architectural influence is the workflow-engine model where durable execution is reconstructed from recorded history rather thanf rom the memory of one running process. Temporal describes this model clearly. Flink is relevant froma  different angle because of its checkpointing, savepoint, and restart semantics.

The concurrency model is influenced by structured concurrency, as reflected in AnyIO and related work. The goal is for Wabizabi's concurrency helpers to reflect runtime structure rather than simply spawn detached work that becomes hard to observe, cancel, or replay.

The current AI orchestration landscape is also part of the prior art. Prefect, Dagster, LangGraph, PydanticAI, CrewAI, and DSPy helped clarify both the patterns worth preserving and the ones worth avoiding. The useful framing is not that those frameworks are wrong and this one is right. Rather, they show the application patterns the field keeps coming back to, and Wabizabi is an attempt to provide a different substrate beneath them.
